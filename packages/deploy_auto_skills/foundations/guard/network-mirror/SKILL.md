# Skill: guard.network-mirror

## Context & Alert

面对缺少基础全局缓存的项目首部部署（尤其是在像 Docker 这种白纸一张的完全隔离镜像环境内加载初始依赖），网络请求的效率常常是验证任务成败的决定性因素。没有设定正确镜像源或缺失网络加速考量的执行流，很容易在 `npm install`、`pip install` 或 `apt-get` 等环节陷入死等、极速劣化并在几十分钟后最终超时，浪费了昂贵的自动化试错和验证时间。这是早期“基础探针忽视网络环境因素”所造成的短板表现。

## Enforcement Rules

1. **被动注入源信息**：如果在未确认当前网络非常畅通的环境下（比如在生成的 Dockerfile 等白盒自动化装载脚本里），只要涉及产生大量网络模块请求的环节（如前端大额依赖安装），一定要**强制被动添加默认镜像源选项**，例如加上 `--registry=https://registry.npmmirror.com` 或显式覆盖 `pip` 到阿里源。
2. **基建前置探测与干预**：如果 Probe 在探针侦测阶段或者早期命令执行中遇到安装进度卡死无返回日志的情况，Agent 必须果断主动中止进程空转。转而重新生成附带镜像加速地址的指令让用户选择或者再次自动执行，替代毫无防备的干等盲试。

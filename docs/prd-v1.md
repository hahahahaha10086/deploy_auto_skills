# PRD v1

项目名称：GitHub 开源项目自动容器化与一键部署平台

产品定位：面向安全研究 / 漏洞挖掘场景的 GitHub 仓库自动分析、自动容器化、自动部署与失败诊断平台。

## 1. 背景与问题

当前在批量挖掘开源项目漏洞时，最大的时间消耗之一不是漏洞分析本身，而是项目部署。不同仓库的技术栈、依赖、启动方式、环境变量和基础服务差异很大，导致手工部署成本高、失败率高、复用性差。

现有一些 Dockerfile 生成器只能辅助生成配置文本，但无法完成“从 GitHub 仓库地址到实际可访问运行环境”的完整闭环，也缺少失败诊断、自动重试、批量任务和结果沉淀能力。

## 2. 产品目标

打造一个最小可用平台，用户输入 GitHub 仓库地址后，系统能够尽量自动完成以下流程：

- 拉取代码
- 识别项目类型与运行方式
- 判断并复用已有容器配置
- 在缺失时生成 `Dockerfile` / `docker-compose.yml`
- 自动执行构建与启动
- 自动进行健康检查
- 输出成功结果或失败原因
- 在失败时提供 AI 推理诊断与修复建议

## 3. 目标用户

- 安全研究员
- 漏洞挖掘工程师
- 需要快速本地复现开源项目的技术人员
- 需要批量评估开源仓库可部署性的团队

## 4. 核心使用场景

1. 用户输入单个 GitHub 仓库地址，希望快速部署并访问项目。
2. 用户批量导入多个 GitHub 仓库，批量查看哪些项目可自动部署成功。
3. 用户面对部署失败的仓库，希望快速知道失败根因与修复方向。
4. 用户希望沉淀一套对常见技术栈可复用的容器化规则与模板。

## 5. 核心价值

- 大幅减少手工部署时间
- 降低不同技术栈项目的部署门槛
- 将部署失败原因结构化沉淀
- 为后续漏洞验证提供统一运行环境
- 支持规则和经验积累，越用越强

## 6. MVP 范围

首版只覆盖最小闭环，不追求覆盖所有项目类型。

MVP 支持：

- 输入 GitHub 仓库地址发起部署任务
- 支持常见公开仓库
- 支持常见单体项目自动识别
  - Node.js
  - Python
  - Java
  - Go
- 支持识别项目是否已有 `Dockerfile` / `docker-compose.yml`
- 支持自动生成基础 Dockerfile
- 支持自动生成基础 `docker-compose.yml`
- 支持 Docker 构建与启动
- 支持基础健康检查
- 支持任务状态、日志、结果展示
- 支持 AI 推理诊断失败原因
- 支持失败后有限次数自动重试
- 支持任务清理

MVP 暂不支持：

- Kubernetes 部署
- 复杂 monorepo 全量自动编排
- 私有仓库复杂认证体系
- 图形界面程序
- Windows-only 部署场景
- 高复杂度中间件集群

## 7. 产品边界

本产品目标是“提升自动部署成功率和效率”，不是保证任意仓库 100% 自动部署成功。

对于复杂项目，系统可以进入 `needs_manual_review` 状态，并输出结构化诊断信息，而不是强行伪成功。

## 8. 核心功能需求

### 8.1 仓库接入

- 用户可输入 GitHub 仓库地址
- 可选指定 branch
- 创建部署任务
- 为任务分配独立工作目录

### 8.2 项目分析

- 扫描仓库文件树
- 识别技术栈、框架、包管理器、构建命令、启动命令、依赖服务
- 判断是否已有容器配置
- 形成项目画像

### 8.3 部署策略决策

- 优先复用已有 `docker-compose.yml`
- 其次复用已有 `Dockerfile`
- 缺失时按规则模板生成
- 不确定场景下调用 AI 决策模块辅助判断
- 输出部署方案与风险提示

### 8.4 配置生成

- 生成或补充 `Dockerfile`
- 生成或补充 `docker-compose.yml`
- 必要时生成 `.env` 占位建议
- 标记配置来源
  - existing
  - template
  - ai
  - hybrid

### 8.5 执行部署

- 调用 `docker build`
- 调用 `docker compose up -d`
- 捕获构建和运行日志
- 控制超时与失败中断

### 8.6 健康检查

- 检查容器状态
- 轮询端口或 HTTP 响应
- 判断部署是否成功
- 输出访问地址和映射端口

### 8.7 失败诊断

- 分析构建日志与运行日志
- 分类错误类型
- 给出根因推断
- 给出修复建议
- 决定是否值得自动重试

### 8.8 任务管理

- 查看任务列表
- 查看任务详情
- 查看日志
- 手动重试
- 清理环境

## 9. AI 推理决策需求

AI 不直接替代规则执行，而是作为结构化决策增强模块。

AI 主要参与：

- 仓库结构理解
- 部署策略选择
- 启动命令与端口推断
- 失败日志诊断
- 修复建议生成

AI 输出必须结构化，至少包括：

- 推荐部署策略
- 风险点
- 置信度
- 错误分类
- 修复建议
- 是否建议自动重试

## 10. 关键页面

- 部署任务创建页
- 任务列表页
- 任务详情页
- 构建日志页
- 运行日志页
- AI 决策结果页
- 规则模板管理页
- 系统设置页

## 11. 核心流程

1. 用户提交 GitHub 仓库地址
2. 系统创建任务并拉取仓库
3. 系统扫描仓库并生成项目画像
4. 系统选择部署策略
5. 系统生成或复用容器配置
6. 系统执行构建与启动
7. 系统健康检查
8. 输出成功结果或失败诊断
9. 需要时执行一次或有限次自动修复重试

## 12. 任务状态定义

- `queued`
- `cloning`
- `analyzing`
- `planning`
- `generating`
- `building`
- `starting`
- `checking`
- `succeeded`
- `failed`
- `needs_manual_review`
- `cleaned`

## 13. 成功标准

MVP 成功不以“覆盖所有仓库”为标准，而以“形成可用闭环”为标准。

首版验收建议：

- 单仓库从输入 URL 到结果输出形成完整闭环
- 对常见单体项目有可接受成功率
- 每个失败任务都能输出可读原因
- 日志、结果、配置来源可追踪
- 环境可一键清理
- 系统支持后续规则和模板积累

## 14. 非功能需求

- 每个任务必须有隔离工作目录
- 每个任务的 Docker 资源命名需唯一
- 必须支持超时控制
- 必须保留任务日志
- 必须支持环境清理
- 必须支持后续批量任务扩展
- 必须可复盘 AI 决策记录

## 15. 风险与限制

- 非标准项目结构可能识别失败
- 缺失环境变量的项目可能无法自动启动
- 多服务复杂依赖项目成功率较低
- AI 可能产生错误推断，因此必须有规则和执行校验兜底
- 自动修复不应无限循环，避免资源浪费和错误放大

## 16. 后续阶段规划

- Phase 1：单仓库自动部署闭环
- Phase 2：AI 失败诊断与自动重试
- Phase 3：规则模板管理与管理后台完善
- Phase 4：批量仓库任务与成功率统计
- Phase 5：复杂项目支持与策略优化

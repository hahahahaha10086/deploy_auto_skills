# 测试方案 v1

## 1. 文档目标

本文档定义 MVP 阶段测试目标、测试范围、测试层级、样本仓库测试集和验收标准。

## 2. 测试目标

- 验证部署链路可完整执行
- 验证常见项目类型的自动部署可用性
- 验证失败诊断与日志可读性
- 验证任务隔离、清理和重试能力
- 验证 AI 决策输出结构完整性

## 3. 测试范围

覆盖以下范围：

- API 接口
- 任务状态机
- 项目分析逻辑
- 模板生成逻辑
- Docker 执行逻辑
- 健康检查逻辑
- 失败诊断逻辑
- 管理后台关键流程

## 4. 测试分层

### 4.1 单元测试

覆盖重点：

- 仓库 URL 校验
- 技术栈识别规则
- 配置模板选择逻辑
- 状态流转逻辑
- 错误分类逻辑
- AI 输出结构校验

### 4.2 集成测试

覆盖重点：

- API -> 编排 -> 分析 -> 生成 -> 执行链路
- 生成配置落盘是否正确
- 日志采集与状态更新是否同步

### 4.3 E2E 测试

覆盖重点：

- 用户从提交仓库到查看结果的完整流程
- 重试流程
- 清理流程
- 查看日志与 AI 决策流程

## 5. 样本仓库测试集

必须准备一组固定样本仓库用于回归测试。

建议样本分类：

- Node.js API 项目
- Node.js 前端项目
- Python Flask 项目
- Python FastAPI 项目
- Java Spring Boot 项目
- Go Web 项目
- 已有 Dockerfile 项目
- 已有 compose 项目
- 缺失环境变量项目
- 启动命令不标准项目
- 多服务依赖项目
- 明显不适合自动部署项目

## 6. 关键测试场景

### 6.1 创建任务

- 合法仓库地址创建成功
- 非法仓库地址报错
- 不存在分支报错

### 6.2 分析阶段

- 正确识别语言和框架
- 正确识别已有 Dockerfile / compose
- 正确识别关键构建文件

### 6.3 策略阶段

- 有 compose 时优先复用
- 有 Dockerfile 时优先复用
- 无容器配置时走模板生成
- 不确定场景正确触发 AI 决策

### 6.4 执行阶段

- 构建成功时状态正确推进
- 构建失败时进入失败诊断
- 启动失败时保留完整日志

### 6.5 健康检查

- HTTP 服务可访问时标记成功
- 非 HTTP 服务能通过端口检测
- 启动超时能正确标记失败

### 6.6 失败诊断

- 缺少依赖能正确归类
- 缺少环境变量能正确归类
- 启动命令错误能正确归类
- 端口冲突能正确归类

### 6.7 清理阶段

- 容器能清理
- 网络能清理
- 卷按策略清理
- 任务状态更新为 `cleaned`

## 7. AI 相关测试

AI 测试不以输出内容完全一致为标准，而以结构和可用性为标准。

验证点：

- 输出字段完整
- 输出可解析
- 置信度字段存在
- 修复建议非空
- 是否建议重试字段正确返回

## 8. 回归测试策略

每次规则或模板调整后，至少执行：

- 单元测试
- 样本仓库回归测试
- 一个成功任务重试验证
- 一个失败任务诊断验证

## 9. 验收标准

MVP 验收建议：

- 样本仓库中的常见单体项目具备可接受成功率
- 每个失败任务都有结构化错误分类
- 后台可查看完整任务链路
- 清理功能可用
- AI 决策输出结构稳定

## 10. 后续增强方向

- 建立标准样本仓库仓
- 建立自动化成功率日报
- 建立错误分类趋势统计
- 引入真实批量任务压测

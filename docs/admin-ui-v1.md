# 管理后台设计 v2

## 1. 文档目标

本文档基于 [prd-v2.md](E:\project\repo-auto-deployer\docs\prd-v2.md) 重新定义管理后台的产品结构、页面层级、核心交互与视觉方向。

本次改版不再把后台理解成“任务列表 + 日志查看器”，而是把它设计成一个面向安全研究场景的“部署作战台”：

- 先判断项目是什么
- 再决定是否值得部署
- 再选择 Skill / AI / 规则路径
- 部署成功后持续管理运行中的靶机

---

## 2. 设计目标

- 让用户在 30 秒内完成单仓库提交与初步判断
- 让“项目分类”成为首屏重点，而不是藏在详情页内部
- 让 Skills 命中情况、AI 参与情况、配置来源一眼可见
- 让成功部署后的容器成为“可管理资产”，而不是一次性结果
- 让批量分析成为前置筛选工具，减少无意义部署

---

## 3. 产品体验定位

### 3.1 界面气质

建议采用“安全控制台 / 作战室”风格，而不是传统 CRUD 后台。

关键词：

- 深色基底 + 高对比信息色
- 明确分层的风险色与状态色
- 强信息密度，但保持强秩序
- 大面积数据面板配合聚焦型详情抽屉
- 像在看“靶场控制中心”，而不是普通管理系统

### 3.2 视觉方向

- 主色：深石墨黑 / 冷灰蓝
- 强调色：酸性绿用于“可运行”，琥珀黄用于“待确认”，高危红用于“失败/崩溃”
- 字体建议：
  - 标题：IBM Plex Sans Condensed / Bebas Neue / DIN Condensed 风格
  - 正文：IBM Plex Sans / Source Sans 3
  - 数据与日志：JetBrains Mono / IBM Plex Mono
- 背景不使用纯色平铺，建议加入低对比网格纹理、轻微噪点、雷达扫描线或拓扑线框

### 3.3 记忆点

后台最强记忆点应是“任务不是卡片，而是目标”。

每个仓库在 UI 中都表现为一个“目标单元”：

- 当前类型
- 可部署性
- Skill 命中
- 风险等级
- 当前运行态

这会让整个产品明显区别于普通部署面板。

## 5. 核心导航结构

### 5.1 总览

面向今天要筛项目、跑环境、查异常的研究员。

### 5.2 单仓库任务

面向“我现在就想把这个 repo 跑起来”的快速路径。

### 5.3 批量分析

面向“先筛 20 个目标，别把时间浪费在不可部署仓库上”的路径。

### 5.4 运行中靶机

面向部署后的持续管理，包括访问、停止、重启、查资源。

### 5.5 Skills

面向知识沉淀与成功率增长，是产品差异化核心。

### 5.6 系统设置

面向运行时策略、AI 配置、资源范围等全局控制。

## 6. 关键页面设计

### 6.1 总览页

目标：

- 让用户快速知道系统当前“可打的目标”有多少
- 看清分类结果、部署成功率、Skill 命中率、运行态健康度

建议模块：

- 今日任务概览
  - 新建任务数
  - 成功部署数
  - `not_deployable` 数
  - `needs_manual_review` 数
- 项目类型分布
  - `web_service`
  - `cli_tool`
  - `library`
  - `desktop_app`
  - `mobile`
  - `complex_service`
  - `unknown`
- Skills 命中看板
  - 命中次数
  - 命中成功率
  - Top Skill
- AI 参与情况
  - 分类使用 AI 次数
  - 方案生成使用 AI 次数
  - 失败诊断使用 AI 次数
- 运行中靶机速览
  - 在线实例数
  - 异常实例数
  - CPU / 内存 Top 5
- 最近任务流
  - 以时间轴方式展示最近 10 条任务

推荐布局：

- 顶部：四到六个高权重 KPI
- 中部左侧：类型分布 / Skills 命中
- 中部右侧：运行中靶机状态墙
- 底部：最近任务时间轴

---

### 6.2 新建任务页

目标：

- 最短路径发起单仓库分析 / 部署

建议字段：

- GitHub 仓库地址
- Branch
- 运行模式
  - 仅分析
  - 分析并自动部署
- 失败后自动重试
- 备注

交互重点：

- 输入 URL 后立即做仓库合法性校验
- 在提交前给出“预估项目类型 / 预估策略”的轻提示
- 提交后默认进入任务详情页，而不是回列表

页面结构建议：

- 左侧：输入表单
- 右侧：系统将如何处理该仓库的说明
  - 分类优先
  - Skill 匹配
  - AI 兜底
  - 不可部署类型会直接跳过

---

### 6.3 任务列表页

目标：

- 批量浏览所有目标
- 快速筛出“值得继续跟进”的任务

表格列建议：

- 任务 ID
- 仓库名
- 项目类型
- 当前状态
- Skill 命中情况
- 配置来源
- 当前阶段
- 更新时间
- 快捷操作

筛选项建议：

- 状态
- 项目类型
- 是否可部署
- Skill 来源
  - history
  - skill
  - ai
  - none
- 关键词
- 时间范围

列表交互建议：

- 默认不是普通表格，而是“高密度目标表”
- 每行附带风险色条
- 点击行右侧直接打开详情侧栏，不强制整页跳转

---

### 6.4 任务详情页

目标：

- 让一个任务从“提交”到“运行 / 失败 / 跳过”的完整链路完全可复盘

建议分成 5 个区块：

1. 顶部任务头
   - 仓库名
   - 当前状态
   - 项目类型
   - 可部署性判断
   - 快速操作按钮

2. 左侧主信息区
   - 生命周期时间线
   - 分类结论
   - Skill 匹配结果
   - 部署策略

3. 右侧诊断区
   - 风险摘要
   - AI 结论
   - 推荐下一步
   - 自动重试建议

4. 底部 Tab 区
   - 生成产物
   - 构建日志
   - 运行日志
   - 健康检查
   - 审计记录

5. 页尾运行态区
   - 当前 endpoint
   - 容器 / 进程状态
   - CPU / 内存
   - 停止 / 重启 / 清理

任务详情中的重点字段：

- `projectCategory`
- `isDeployable`
- `detectedFramework`
- `matchedSkill`
- `strategySource`
- `envHints`
- `errorCategory`
- `shouldRetry`

---

### 6.5 分类与 Skill 匹配区

这是 v2 新增的核心可视化模块，建议从旧版详情页中单独强化出来。

建议展示：

- 项目类型
- 是否可部署
- 分类依据
- 置信度
- 命中的 Skill
- Skill 来源
  - built_in
  - auto_learned
  - user_defined
- Skill 成功率
- 未命中时的 AI 兜底说明

可视化形式建议：

- 左边显示“分类卡”
- 右边显示“Skill 命中卡”
- 两张卡中间用箭头或流程线连接

让用户直观看到：

“这个仓库先被判定成什么，再走的哪套知识包”

### 6.6 构建日志与运行日志

目标：

- 真正支持问题定位，而不是只展示纯文本

构建日志页建议：

- 支持按阶段切换
  - clone

  - skill_match

  - start
  - healthcheck
- 支持关键词搜索
- 支持错误行高亮

### 6.7 AI 诊断结果页

目标：

- 让 AI 输出像“结构化分析结论”，而不是聊天记录

建议字段：

- 根因类别
- 根因描述
- 修复建议
- 是否建议自动重试

### 6.8 批量分析页

目标：

- 在正式部署前快速做仓库筛选

输入方式建议：

- 文本批量粘贴 URL
- 上传 txt / csv
- 每行一个仓库地址

运行模式：

- 仅分类
- 分类 + 预估可部署性
- 分类 + Skill 匹配预览

输出字段建议：

- 仓库地址
- 项目类型
- 是否可部署
- 命中 Skill
- 预估成功率
- 风险点
- 建议动作

  - 人工复核
  - 直接跳过

### 6.9 批量结果页

目标：

- 让用户一眼筛出“今天值得打的 repo”

建议能力：

- 按预估成功率排序
- 按项目类型分组

- 一键忽略不可部署项目

视觉建议：

- 不要做普通表格

- 高成功率目标放在顶部，颜色更亮
- `not_deployable` 目标进入折叠区，降低视觉干扰

建议分组：

## 7. 关键交互流程

### 7.1 单仓库任务流程

1. 用户输入 GitHub URL
2. 系统立即校验 URL
3. 创建任务后进入详情页
4. 页面先显示 `classifying`
5. 再显示 `skill_matching`
6. 若 `not_deployable`，详情页直接给出跳过原因
7. 若可部署，继续进入构建、启动、检查
8. 成功后任务从 `succeeded` 进入 `running`

### 7.2 批量分析流程

1. 用户粘贴 10 至 50 个 URL
2. 系统仅做分类和 Skill 匹配
3. 输出批量结果
4. 用户勾选值得部署的目标
5. 统一发起部署

### 7.3 运行中靶机管理流程

1. 用户进入“运行中靶机”
2. 查看在线实例
3. 对某个实例执行停止 / 启动 / 查看日志
4. 异常实例可一键打开对应任务详情

## 8. 状态可视化规范

建议统一状态颜色：

- `queued` / `classifying` / `skill_matching` / `planning`
  - 冷蓝色
- `building` / `starting` / `checking`
  - 琥珀色
- `succeeded` / `running`
  - 亮绿色
- `stopped`
  - 中性灰
- `failed` / `crashed`
  - 红色
- `needs_manual_review`
  - 紫红或橙红
- `not_deployable`
  - 石板灰
- `cleaned`
  - 低饱和灰绿

状态标签必须同时包含：

- 颜色
- 图标
- 文本

不能只依赖颜色表达。

## 11. 与旧版设计相比的主要变化

- 从“模板管理”切换为“Skills 中台”
- 从“部署优先”切换为“分类优先”
- 新增 `not_deployable`、`running`、`stopped`、`crashed` 等运行态心智
- 新增批量分析，不再要求所有 URL 都走部署流程
- 新增运行中靶机页，把成功部署后的实例纳入持续管理
- 强化 AI 与规则来源可视化，避免黑盒感

## 13. 结论

后台的核心任务已经不是“看任务有没有跑完”，而是：

- 判断 repo 是什么
- 判断值不值得部署
- 判断命中了哪套 Skill
- 判断部署后的实例现在是否仍然可用

因此，新的后台设计必须围绕以下三个中心展开：

- 分类中心
- 知识中心
- 运行中心

只有这样，这个产品才真正符合 v2 PRD 的定位，而不是停留在 v1 的“部署任务面板”阶段。

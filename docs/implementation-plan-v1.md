# 实施计划 v1

## 1. 文档目标

本文档定义项目从 0 到 MVP 的推荐实施路径、阶段目标、交付物、风险点和验收口径。

## 2. 实施原则

- 先文档，后开发
- 先闭环，后扩展
- 先规则，后 AI 深化
- 先单仓库，后批量任务

## 3. Phase 划分

### Phase 0：项目准备

目标：

- 完成产品文档和技术方案
- 确定技术选型和目录结构
- 初始化仓库和基础规范

交付物：

- PRD
- 架构文档
- 模块职责文档
- API 文档
- 管理后台设计文档
- 测试方案文档

### Phase 1：单仓库最小闭环

目标：

- 输入 GitHub 仓库地址
- 完成 clone、分析、生成、构建、启动、验活
- 在后台查看任务结果

交付物：

- 创建任务接口
- 任务列表和详情页
- 仓库分析模块
- 基础模板生成模块
- Docker 执行模块
- 健康检查模块

验收标准：

- 至少对一批常见简单项目形成完整闭环

### Phase 2：失败诊断与 AI 决策

目标：

- 引入 AI 做策略辅助和失败诊断
- 实现有限自动重试

交付物：

- AI 策略决策模块
- AI 失败诊断模块
- 重试机制
- AI 结果展示页

验收标准：

- 失败任务可以输出结构化诊断

### Phase 3：模板与规则完善

目标：

- 丰富技术栈模板
- 引入模板管理与系统设置

交付物：

- 模板管理页
- 系统设置页
- 更多语言/框架模板

### Phase 4：批量任务

目标：

- 支持批量导入仓库
- 支持任务统计与筛选

交付物：

- 批量任务接口
- 批量任务视图
- 统计页面

### Phase 5：稳定性与复杂项目增强

目标：

- 支持更多复杂场景
- 优化成功率
- 提升可观测性

交付物：

- 复杂仓库策略增强
- 更完善的错误分类
- 更多运行隔离措施

## 4. 推荐里程碑

- M1：完成文档与项目初始化
- M2：完成单仓库闭环
- M3：完成 AI 失败诊断
- M4：完成模板管理和系统设置
- M5：完成批量任务

## 5. 主要风险

- 不同技术栈项目差异大，规则识别准确率不足
- 纯 AI 推理稳定性不够
- Windows 环境下 Docker 行为与 Linux 存在差异
- 自动修复可能导致错误放大

## 6. 风险缓解

- 采用规则优先策略
- 保留完整日志和人工 review 通道
- 控制自动重试次数
- 从固定样本仓库开始扩展

## 7. 建议开发顺序

1. 初始化项目骨架
2. 定义数据模型
3. 实现任务状态机
4. 实现仓库分析模块
5. 实现模板生成模块
6. 实现执行与验活模块
7. 实现后台任务页面
8. 引入 AI 决策模块
9. 完善模板与统计

# 总体架构设计 v1

## 1. 文档目标

本文档定义 GitHub 开源项目自动容器化与一键部署平台的总体架构，明确系统边界、核心模块、任务流转、AI 决策接入点与基础运行模型，作为后续详细设计与开发实施的基础。

## 2. 设计目标

- 支持从 GitHub 仓库地址到容器化部署结果的完整闭环
- 优先提升常见开源项目的自动部署成功率
- 支持批量任务扩展
- 支持失败诊断与有限自动修复
- 保证任务隔离、日志可追踪、结果可复盘
- 保留规则优先、AI 增强的架构演进能力

## 3. 设计原则

### 3.1 规则优先，AI 增强

系统优先基于明确规则和仓库事实做判断。AI 仅在规则不足、不确定性较高或失败诊断场景下参与推理决策。

### 3.2 流水线分阶段执行

部署任务必须拆分为清晰阶段，避免把识别、生成、执行、诊断混在同一层中，方便重试、观测与审计。

### 3.3 结果可复盘

任何一次部署任务都必须可回看：输入仓库、生成配置、执行日志、AI 判断、失败结论都需要留痕。

### 3.4 任务隔离

每个仓库部署任务必须具备独立工作目录、独立 compose project name、独立日志路径，避免任务之间互相污染。

## 4. 系统上下文

用户通过 Web 管理后台或 API 提交 GitHub 仓库地址，平台内部执行以下链路：

1. 创建部署任务
2. 拉取仓库
3. 分析项目结构
4. 决策部署策略
5. 生成或复用容器配置
6. 执行 Docker 构建与启动
7. 健康检查
8. 失败诊断和有限重试
9. 输出最终结果

系统与以下外部依赖交互：

- GitHub
- Docker Desktop / Docker Engine
- LLM 服务提供商
- 本地文件系统
- 数据库

## 5. 逻辑架构

系统由以下核心层组成：

- 展示层
- API 接入层
- 任务编排层
- 仓库分析层
- 配置生成层
- AI 决策层
- 执行与验活层
- 日志与审计层
- 数据持久化层

### 5.1 展示层

负责部署入口、任务列表、任务详情、日志展示、AI 决策结果与系统设置页面。

### 5.2 API 接入层

负责参数校验、任务创建、任务查询、任务重试、清理、规则模板管理等接口能力。

### 5.3 任务编排层

负责维护任务状态机、控制阶段流转、超时、重试策略和失败终止条件，是系统主调度核心。

### 5.4 仓库分析层

负责 Git 仓库拉取、文件树扫描、技术栈识别、配置文件识别、项目画像生成。

### 5.5 配置生成层

负责复用已有容器配置、规则模板生成、AI 辅助补全与结果落盘。

### 5.6 AI 决策层

负责不确定项目的策略推断、失败日志诊断、修复建议和是否自动重试判断。

### 5.7 执行与验活层

负责执行 `docker build`、`docker compose up`、容器状态检查、HTTP/TCP 健康检查、日志采集与清理。

### 5.8 日志与审计层

负责记录任务阶段日志、构建日志、运行日志、AI 输入输出摘要、操作审计。

### 5.9 数据持久化层

负责任务信息、项目画像、生成配置、执行结果、AI 决策记录、规则模板与系统设置存储。

## 6. 任务状态机

任务状态定义如下：

- `queued`
- `cloning`
- `analyzing`
- `planning`
- `generating`
- `building`
- `starting`
- `checking`
- `succeeded`
- `failed`
- `needs_manual_review`
- `cleaned`

推荐状态流转如下：

1. `queued` -> `cloning`
2. `cloning` -> `analyzing`
3. `analyzing` -> `planning`
4. `planning` -> `generating`
5. `generating` -> `building`
6. `building` -> `starting`
7. `starting` -> `checking`
8. `checking` -> `succeeded` 或 `failed`
9. `failed` -> `planning` 或 `needs_manual_review`

## 7. 核心执行流程

### 7.1 创建任务

- 接收仓库地址与可选参数
- 创建任务记录
- 分配任务 ID、工作目录、日志目录

### 7.2 拉取仓库

- 校验仓库地址合法性
- `git clone`
- 支持指定分支
- 记录 clone 结果

### 7.3 项目分析

- 扫描目录结构
- 识别关键文件
- 推断语言、框架、包管理器
- 识别已有 Dockerfile 和 compose
- 生成项目画像

### 7.4 部署策略决策

- 规则判断优先级
- 必要时调用 AI 推理
- 输出推荐策略与风险点

### 7.5 配置准备

- 复用已有配置
- 缺失时使用模板生成
- 不确定时由 AI 补全建议
- 形成最终执行配置

### 7.6 构建与启动

- 执行镜像构建
- 执行 compose 启动
- 采集构建日志与容器日志

### 7.7 健康检查

- 检查容器是否存活
- 检查端口连通性
- 检查 HTTP 响应
- 输出访问地址

### 7.8 失败处理

- 分类失败阶段
- 调用 AI 诊断
- 判断是否可自动修复
- 控制重试次数

## 8. AI 决策接入点

AI 参与但不主导执行，主要接入以下位置：

### 8.1 项目策略判断

输入：

- 文件树摘要
- 关键文件摘要
- README 摘要
- 初步规则识别结果

输出：

- 项目类型
- 推荐部署策略
- 启动命令建议
- 端口建议
- 服务依赖建议
- 风险点
- 置信度

### 8.2 失败诊断

输入：

- 失败阶段
- 构建日志摘要
- 容器日志摘要
- 当前生成配置
- 项目画像

输出：

- 错误分类
- 根因推断
- 修复建议
- 是否自动重试
- 置信度

```text
  artifacts/
    <task-id>/
      Dockerfile

      strategy.json
  logs/
    <task-id>/
      task.log
      build.log
      compose.log
      healthcheck.log
```

## 10. 技术选型建议

MVP 建议选型如下：

- Web：React
- API：Node.js + Fastify 或 Express
- 任务队列：MVP 可用本地串行或内存队列，后续升级 Redis + BullMQ
- 数据库：SQLite
- 执行层：Node 子进程调用 `git` 和 `docker`
- AI Provider：可插拔抽象层
- 日志：文件日志 + 数据库存储摘要

## 11. 安全与隔离

- 每个任务独立工作目录
- 每个任务独立 Docker Compose project name
- 端口冲突检测
- 超时自动终止
- 任务结束支持清理容器、网络、卷
- 默认限制自动重试次数
- 保留后续资源限制扩展能力

## 12. 可观测性要求

- 每阶段必须记录开始时间、结束时间、状态和结果
- 每次 AI 调用必须保留输入摘要、输出摘要和置信度
- 每次执行命令必须记录 stdout/stderr 摘要
- 管理后台必须可查看任务全生命周期

## 13. 演进方向

- 从单仓库任务演进到批量任务
- 从规则模板演进到规则库
- 从失败诊断演进到有限自动补丁
- 从 SQLite 演进到更适合并发场景的数据库和消息队列
